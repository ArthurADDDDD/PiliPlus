# PiliPlus 功耗/发热测试指南（Pixel 10 Pro）

配合 [`tools/monitor_piliplus_power.py`](../tools/monitor_piliplus_power.py) 使用。
本指南只描述**怎么测、怎么分清热源**，不包含任何已完成的测试结论——
所有测试都需要在真机（Pixel 10 Pro）上手动执行，本仓库/工具本身不会自动验证省电效果。

## 目的

用户反馈：PiP 听歌场景下手机仍然发热、耗电，但不确定主要负载来自：

- 屏幕常亮
- 视频硬解码
- PiP 合成
- 字幕渲染
- 弹幕渲染
- Flutter 持续渲染
- 网络/播放器其他逻辑

本指南给出一组**受控变量**的测试组，通过两两对比隔离出每种因素的增量成本。

## 测试前置条件（每组测试都要满足）

- 同一个视频、同一个时间段（seek 到同一起点，播放同样长度）
- 同一清晰度（建议固定为常用档位，如 1080P，不要开自动切换）
- 同一网络（同一 Wi-Fi 或同一蜂窝网络，不要中途切换）
- 同一音量
- 固定屏幕亮度（关闭自动亮度调节，手动设置一个固定值）
- 固定刷新率设置（同一档 `Android 17/Tensor 省电刷新率` 开关状态；两次对比要测的
  变量不是这个开关本身时，两组都保持一致）
- 相近的初始电量（建议每组开始前电量差不超过 3%）
- **手机先静置降温**：测试前让手机静置至少 5～10 分钟，确认电池温度回落到
  接近室温/基线水平再开始下一组（可用 `dumpsys battery` 或本工具的
  `battery_temp_c` 观察）
- 每组建议至少 15～20 分钟（`--duration 1200` 起步，越长越能看出稳定后的斜率）
- 组与组之间留足够间隔恢复温度（不要背靠背连续测试，温度基线会累积漂移）

## 测试矩阵

| # | 场景 | 说明 |
|---|------|------|
| 1 | PiP 播放，字幕关，弹幕关 | 基线：PiP 合成 + 解码 + 亮屏，无叠加层 |
| 2 | PiP 播放，字幕开，弹幕关 | 字幕叠加层的增量成本 |
| 3 | PiP 播放，字幕关，弹幕开 | 弹幕叠加层的增量成本；**仅在该 PiP 路径支持弹幕时测试**（本 fork 当前返回键原生 PiP 不渲染弹幕，若测试该路径可跳过 #3/#4；回桌面 PiP 路径保留弹幕，可在该路径下测试） |
| 4 | PiP 播放，字幕开，弹幕开 | 字幕+弹幕叠加的综合成本 |
| 5 | 相同视频正常全屏播放 | 无 PiP 合成、无常亮限制之外的额外负载，作为“正常使用”基线 |
| 6 | 相同内容后台/锁屏只听声音 | 屏幕关闭、无视频渲染，只有音频解码 + 网络 |
| 7 | PiP 暂停但屏幕保持亮 | 静态画面 + 亮屏 + PiP 窗口合成，但无持续解码，用于估计“亮屏 + 静态合成”基线 |

> 字幕（subtitle）和弹幕（danmaku）是两套完全不同的渲染路径，**不要混为一谈**：
> 字幕是 mpv/libass 内部渲染进视频帧，弹幕是 Flutter 在视频层上叠加的独立
> 图层。分别控制、分别测试，才能把两者的成本拆开。

## 如何解读对比结果

用 `compare` 子命令两两对比后，按下表方式解读差值（都基于同一时长下的
平均/中位绝对电流、估算功率、温升斜率）：

| 对比 | 反映的增量 |
|------|-----------|
| #1 vs #2 | 字幕渲染的额外成本 |
| #1 vs #3 | 弹幕渲染的额外成本 |
| #1 vs #6 | 视频解码 + PiP 合成 + 亮屏的综合额外成本（相对纯后台听音频） |
| #1 vs #7 | 持续视频解码与播放（相对暂停/静态画面）的额外成本 |
| #1 vs #5 | PiP 合成本身相对全屏播放的额外/减少成本（会受屏幕整体亮度区域不同影响，谨慎解读） |

**重要**：

- 单轮测试的噪声很大（环境温度、后台任务、网络抖动都会影响），
  **不要根据单轮结果下结论**。同一组场景建议重复至少 2～3 轮，
  确认多轮结果方向一致后再比较。
- 温升斜率（C/min）比总温升更适合比较**时长不同**的测试组。
- 电流的原始符号（`current_now_sign`）在不同设备/内核上约定不一致，
  本工具只用绝对值估算功率，**不代表已校准的真实功率**，仅作相对比较参考。
- 短时间（15～20 分钟）测试**不能直接等价于完整续航表现**——只能用来做
  相对比较（"这个开关让短期功耗方向性地更高/更低"），不能换算成
  "一小时省多少电" 这类结论。

## 操作步骤

1. 手机静置降温，确认 `dumpsys battery` 温度接近基线。
2. 按测试矩阵设置好视频/字幕/弹幕/清晰度/亮度/刷新率状态。
3. 开始录制（示例：场景 1，PiP 无字幕无弹幕）：

   ```powershell
   python tools\monitor_piliplus_power.py record `
     --label pip-no-overlay `
     --duration 1200 `
     --interval 10
   ```

4. 立即在手机上进入对应场景（进入 PiP / 打开字幕 / 打开弹幕等），
   让场景状态在整个录制窗口内保持稳定。
5. 录制自然结束（达到 `--duration`）或按 Ctrl+C 提前结束——
   两种方式都会正常写出 `end/` 快照和 `SUMMARY.md`。
6. 重复步骤 1～5，测完矩阵中其余场景（每个场景建议重复 2～3 轮，
   标签里可以加轮次后缀，如 `pip-no-overlay-r1` / `pip-no-overlay-r2`）。
7. 用 `compare` 对比想看的两组：

   ```powershell
   python tools\monitor_piliplus_power.py compare `
     .review\power\*-pip-no-overlay-* `
     .review\power\*-pip-subtitle-on-*
   ```

8. 阅读输出表格和「Comparability caveats」部分——工具会主动标出
   时长不同、起始温度不同、缺采样率过高等不可直接比较的情况。

## 输出目录结构

每次 `record` 在 `.review/power/<时间戳>-<label>/` 下生成：

- `samples.csv`：逐条采样（电池、电流、功率估算、CPU、内存、屏幕、
  刷新率、PiP 状态、前台 Activity、热传感器、decoder 提示、错误计数）
- `metadata.json`：本次运行的参数、设备信息、实际时长、是否被中断等
- `start/`、`end/`：开始/结束时的原始 `dumpsys` 快照
  （battery / thermalservice / hardware_properties / display /
  activity_activities / media.metrics / media.codec / gfxinfo /
  meminfo / batterystats）
- `logcat.txt`：过滤后的 logcat（MediaCodec / Surface / EGL / PipShell /
  崩溃 / 热相关行，以及提及包名的行）
- `SUMMARY.md`：本轮的统计摘要，含明确的“短时间测试不能等价续航”声明

## 已知限制

- 工具不需要 root，但也因此拿不到部分需要特殊权限的字段
  （例如某些设备上 `current_now` 需要额外权限）——这些字段会被
  明确标记为 `unavailable`，不会伪造数值。
- 电流符号约定因设备而异，工具**不会**据此判断充电/放电方向；
  充电状态请看 `battery_status`（来自 `dumpsys battery` 的 `status` 字段）。
- decoder 识别（`codec_hint`）是启发式匹配 `dumpsys media.metrics` 文本，
  仅供参考，不是权威的“当前正在用哪个 codec”结论。
- PiP 状态检测（`in_pip`）同样是启发式匹配 `dumpsys activity activities`，
  不同 Android 版本字段可能有差异。
